# 课堂演示脚本

## 开场介绍

本项目是一个基于 Ollama 本地大模型和 RAG 技术的网站智能客服系统。它面向 NXP AIoT Cloud、Ara240 DNPU、Edge AI、LLM Edge Studio 等领域资料，目标是展示从知识库检索到本地模型生成回答的完整链路。

## 系统架构讲解

先展示页面左侧状态栏：本地模型连接状态、LLM 模型、Embedding 模型、知识库文档数、chunk 数和向量库类型。

再说明后端结构：FastAPI 接收问题，RAG 模块负责向量化和相似度检索，Ollama 负责本地生成回答，前端展示回答和来源。

## 主演示问题

```text
我想基于 NXP AIoT Cloud 搭建一个离线运行的网站智能客服系统，用来回答 Ara240、LLM Edge Studio 和边缘 AI 部署相关问题。请问这个系统应该采用什么整体架构？知识库怎么构建？用户提问后 RAG 的完整流程是什么？本地部署相比云端调用有什么优势？
```

## 演示步骤

1. 打开 `http://localhost:8000`。
2. 点击左侧“重建知识库索引”，说明系统会读取 `data/knowledge_base` 并构建向量索引。
3. 点击主演示问题，让问题自动填入输入框。
4. 点击发送。
5. 观察右侧六步 RAG 流程：接收问题、问题向量化、知识库检索、Prompt 构造、本地模型生成、返回答案与来源。
6. 展开检索来源，说明每条来源都有标题、类别、相似度、来源和内容片段。
7. 阅读回答末尾的参考资料，强调回答基于本地知识库。

## 重点说明

- 本项目不直接让大模型自由发挥，而是先检索知识库再生成回答。
- 检索来源可视化可以提高可解释性，适合网站客服和课程展示。
- 本地 Ollama 可以减少云端依赖，适合离线实验、隐私敏感和边缘端场景。
- TF-IDF 和 SimpleVectorStore 是 fallback，保证没有完整环境时也能演示。

## 备用问题

- 用户提问后，RAG 智能客服系统是如何从知识库中找到资料并生成回答的？
- LLM Edge Studio 和普通云端大模型调用有什么区别？它为什么适合边缘端智能应用？
- Ara240 DNPU 在边缘 AI 视觉分析任务中主要承担什么作用？
- 为什么这个项目不直接使用通用大模型回答，而要加入本地知识库和 RAG 检索？

## 现场翻车时的备用讲法

如果 Ollama 没启动或模型没拉取，可以说：系统已经完成了知识库检索和来源展示，但本地模型服务当前不可用。为了保证课堂演示，可以在 `.env` 中设置 `MOCK_LLM=true` 和 `EMBED_PROVIDER=tfidf`，这会用本地 mock 回答和 TF-IDF 检索演示完整流程，不依赖外部服务。

如果 ChromaDB 不可用，可以说：系统自动退回到 JSON 文件保存向量，并用 numpy 余弦相似度检索。这体现了项目的可运行性设计。
